# OpenClaw Spec v0

**Status: Draft, Q2 2026. Not implementation guidance yet.**

This document sketches the minimal OpenClaw agent surface. Subject to
change before Q3 2026 alpha.

## Agent definition

An OpenClaw agent is defined by four elements:

1. A **W3C DID** (`did:ainfera:agent:{uuid}`) as canonical identity
2. An **MCP server** describing its tools
3. An **A2A agent card** describing its capabilities to peers
4. An **Ainfera SDK hook** for trust scoring and metered billing

## Trust integration

Every OpenClaw agent emits execution telemetry to Ainfera's ATS engine:

- **Reliability** — pulled from uptime, latency, error rate
- **Quality** — pulled from user feedback, groundedness checks,
  NeMo Guardrails outputs
- **Cost-efficiency** — pulled from cost-per-task vs. benchmark
- **Security** — pulled from prompt injection tests, credential leak
  scans, NeMo Guardrails safety signals
- **Compliance** — pulled from audit log completeness, regulatory
  posture (SOC 2, PDPA, GDPR)

The geometric mean of the five dimensions produces the overall ATS score
(0–1000). Any dimension at zero collapses the overall to zero. There is
no gaming this.

## A2A communication

Agents communicate via the Agent2Agent protocol:

```python
from openclaw import discover

# Find a trusted agent by capability
agent = discover(capability="document-summarization", min_trust=800)

# Invoke it as a peer
result = await agent.invoke({"document": "..."})
```

## Kill switch

Every agent is armed with a kill switch on deployment. Any of the
following triggers immediate quarantine (sub-500ms propagation):

- Manual trigger via CLI (`ainfera kill <agent-id>`)
- Manual trigger via API (`POST /v1/agents/{id}/kill`)
- Trust score drops below `auto_kill_below` threshold (default: 400)
- Anomaly detector flags high-severity event (hallucination, injection,
  exfiltration)

## Policy engine (Stage 1 — Q4 2026)

A2A agent calls will route through a Cedar + OPA policy engine:

- Agent A wants to invoke Agent B
- Policy engine checks Agent A's credentials, Agent B's access list,
  trust thresholds, rate limits, and compliance constraints
- If authorized: call proceeds with an audit entry
- If not: denial with structured reason

Policies will be authored in Cedar, evaluated via OPA. More detail in
spec-v1.

## Open questions

These are explicit gaps in v0 that v1 must address:

1. **Long-running tasks.** How does OpenClaw handle multi-hour workflows?
   Streaming? Checkpointing? Idempotency?
2. **LangChain/CrewAI migration.** What does the shim layer look like?
   Is it an import-compatible wrapper, or a structured rewrite tool?
3. **Multi-tenant isolation.** How does the policy engine scale to
   millions of agents owned by thousands of customers?
4. **Sandbox escape.** Docker sandboxing is the baseline; Firecracker
   microVMs are planned for Stage 2. Under what threat model is each
   sufficient?
5. **Cost-efficiency dimension.** How is "cost per task vs. benchmark"
   defined in a way that doesn't penalize agents for cold starts or
   variable input sizes?

Open an Issue with opinions.
